# Security Policy

## Supported Versions

As versões abaixo indicam quais releases do projeto recebem correções de segurança.

| Version                        | Supported |
| ------------------------------ | --------- |
| Latest / main                  | ✅         |
| Releases estáveis recentes     | ✅         |
| Versões antigas sem manutenção | ❌         |

## Reporting a Vulnerability

Se você encontrou uma vulnerabilidade de segurança neste projeto, por favor, reporte de forma responsável.

### Como reportar

Você pode abrir uma issue privada de segurança no GitHub, se o recurso estiver disponível no repositório, ou entrar em contato diretamente pelo e-mail:

**[erickigaudencio@gmail.com](mailto:erickigaudencio@gmail.com)**

Ao reportar, inclua o máximo de detalhes possível:

* Descrição da vulnerabilidade;
* Passos para reproduzir o problema;
* Arquivos, scripts ou partes do sistema afetadas;
* Possível impacto;
* Prints, logs ou vídeos, se houver;
* Sugestão de correção, caso tenha.

### O que acontece depois do reporte

Após o envio do relatório:

1. O problema será analisado.
2. Caso a vulnerabilidade seja confirmada, uma correção será planejada.
3. Dependendo da gravidade, a correção poderá ser priorizada antes de novas funcionalidades.
4. Após a correção, uma nova versão ou commit de segurança poderá ser publicado.
5. Se o problema não for considerado uma vulnerabilidade, será explicado o motivo.

### Tempo de resposta

O objetivo é responder relatórios de segurança em até **7 dias**.

Correções críticas podem ser tratadas com prioridade maior, dependendo do impacto no projeto.

## Responsible Disclosure

Por favor, não divulgue publicamente a vulnerabilidade antes que ela seja analisada e corrigida.

Isso ajuda a proteger usuários, colaboradores e qualquer pessoa que utilize este projeto.

## Security Best Practices

Para manter o projeto seguro:

* Não envie senhas, tokens, chaves de API ou dados sensíveis para o repositório;
* Use variáveis de ambiente para informações privadas;
* Revise dependências antes de adicioná-las;
* Mantenha bibliotecas e pacotes atualizados;
* Evite executar scripts desconhecidos sem revisão;
* Faça backup antes de aplicar alterações críticas.

## Scope

Esta política se aplica ao código, arquivos de configuração, scripts, assets e qualquer componente incluído oficialmente neste repositório.

Problemas fora deste repositório ou causados por modificações de terceiros podem não ser cobertos por esta política.
